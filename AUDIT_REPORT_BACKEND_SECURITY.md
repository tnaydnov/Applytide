## SCAN 1: Pydantic Models with Loose/Dangerous Types

### Finding 1.1 — `Dict[str, Any]` fields throughout document schemas (MEDIUM)

**Files & Lines:**
- `backend/app/api/schemas/documents.py` L44: `template_data: Dict[str, Any]`
- `backend/app/api/schemas/documents.py` L54-59: `job_match_summary`, `keyword_analysis`, `missing_skills`, `tech_analysis`, `ai_detailed_analysis`, `section_quality` — all `Dict[str, Any]`
- `backend/app/api/schemas/documents.py` L87: `metadata: Dict[str, Any]`

**Problem:** These are **response-only** models (`DocumentTemplate`, `DocumentAnalysis`, `DocumentResponse`) generated by the server. While `Dict[str, Any]` on response models is less dangerous than on input models, `metadata` and `template_data` could flow back through APIs that accept them. If any endpoint round-trips these values back to the server, arbitrary nested data can be injected.

**Severity:** MEDIUM  
**Fix:** For response-only fields, this is acceptable. Verify no endpoint accepts `DocumentResponse.metadata` directly as input. If any do, constrain the value types or max depth.

---

### Finding 1.2 — `Dict[str, Any]` on INPUT schemas: preferences (MEDIUM)

**Files & Lines:**
- `backend/app/api/schemas/preferences.py` L14: `preference_value: Dict[str, Any]` in `PreferenceCreate`
- `backend/app/api/schemas/preferences.py` L24: `preference_value: Dict[str, Any]` in `PreferenceUpdate`

**Problem:** The `preference_key` is validated against an allow-list (good), but `preference_value` accepts **any** nested structure. A client could send deeply nested objects (`{"a":{"b":{"c":...}}}` × 1000 levels) causing JSON parsing overhead or store massive blobs.

**Severity:** MEDIUM  
**Fix:**
```python
from pydantic import field_validator
import json

class PreferenceCreate(BaseModel):
    preference_key: str = Field(...)
    preference_value: Dict[str, Any]

    @field_validator('preference_value')
    @classmethod
    def validate_value_size(cls, v):
        serialized = json.dumps(v)
        if len(serialized) > 10_000:  # 10KB max
            raise ValueError("Preference value too large")
        return v
```

### Finding 1.3 — `Dict[str, Any]` on INPUT: reminder notification_schedule (MEDIUM)

**Files & Lines:**
- `backend/app/api/schemas/reminders.py` L26: `notification_schedule: Optional[Dict[str, Any]] = None` in `ReminderCreate`

**Problem:** Same issue — arbitrary JSON with no schema enforcement or size limit. If this is stored directly in the database, a malicious user could store megabytes of arbitrary JSON.

**Severity:** MEDIUM  
**Fix:** Define a typed schema:
```python
class NotificationSchedule(BaseModel):
    minutes_before: Optional[int] = Field(None, ge=0, le=10080)  # max 1 week
    email: bool = False
    push: bool = False

class ReminderCreate(ReminderBase):
    notification_schedule: Optional[NotificationSchedule] = None
```

### Finding 1.4 — `Dict[str, Any]` on INPUT: AI extraction fields (MEDIUM)

**Files & Lines:**
- `backend/app/api/routers/ai.py` L44-48: `ExtractIn` model has 5 `Dict[str, Any]` fields:
  - `jsonld: Optional[List[Dict[str, Any]]]`
  - `metas: Optional[Dict[str, Any]]`
  - `readable: Optional[Dict[str, Any]]`
  - `xhrLogs: Optional[List[Dict[str, Any]]]`
  - `quick: Optional[Dict[str, Any]]`

**Problem:** The `html` field has a size check later (`manual_text` capped at 200KB), but **none** of these dict/list fields have size limits. A client could send a 100MB `jsonld` array or deeply nested `metas` dict, causing memory exhaustion or slow processing.

**Severity:** MEDIUM  
**Fix:** Add a top-level validator that checks total serialized payload size:
```python
from pydantic import model_validator
import json

class ExtractIn(BaseModel):
    # ... fields ...

    @model_validator(mode='after')
    def check_total_size(self):
        total = len(json.dumps(self.model_dump(), default=str))
        if total > 2_000_000:  # 2MB max total
            raise ValueError("Request payload too large")
        return self
```

### Finding 1.5 — URL fields without URL validation (LOW)

**Files & Lines:**
- `backend/app/api/schemas/auth.py` L131-133: `ProfileUpdateIn` fields `website`, `linkedin_url`, `github_url` are plain `str` with max_length only
- `backend/app/api/schemas/auth.py` L76: `GoogleOAuthRegisterIn.avatar_url` — plain `str`

**Problem:** These fields accept any string, including `javascript:`, `data:`, or file:// URIs. If rendered in the frontend without additional sanitization, they could be used for stored XSS.

**Severity:** LOW  
**Fix:**
```python
from pydantic import HttpUrl

class ProfileUpdateIn(BaseModel):
    website: Optional[HttpUrl] = None
    linkedin_url: Optional[HttpUrl] = None
    github_url: Optional[HttpUrl] = None
```
Or add a validator that enforces `https://` prefix.

### Finding 1.6 — `extra: Optional[dict]` in error payload (LOW)

**File:** `backend/app/api/routers/errors.py` L58  
**Line:** `extra: Optional[dict] = None`

**Problem:** The `FrontendErrorPayload.extra` is `Optional[dict]` (untyped). This is a **public endpoint** (no auth required). An attacker could send massive arbitrary JSON blobs that get stored directly in the database (`ApplicationLog.extra`). This is a potential storage DoS vector.

**Severity:** MEDIUM  
**Fix:** Add size validation and constrain the type:
```python
extra: Optional[Dict[str, str]] = Field(None, max_length=20)  # max 20 keys

@field_validator('extra')
@classmethod
def validate_extra(cls, v):
    if v and len(json.dumps(v)) > 5000:
        raise ValueError("Extra data too large")
    return v
```

## SCAN 2: Endpoints Accepting Raw dict/JSON Without Validation

### Finding 2.1 — `attach_from_document` accepts raw `dict` (HIGH)

**File:** `backend/app/api/routers/applications/attachments.py` L33  
**Line:** `payload: dict,`

**Problem:** This endpoint parameter is typed as `dict` — **completely bypassing Pydantic validation**. FastAPI will parse any JSON body and pass it through. The code then accesses `payload.get("document_id")` and `payload.get("document_type")` without any type checking, length limits, or required-field enforcement.

An attacker can:
- Omit `document_id` entirely (it defaults to `None`, passed to `str(None)` = `"None"`)
- Send arbitrary extra keys that pollute logs
- Send non-string values for `document_type`

**Severity:** HIGH  
**Fix:** Create a typed Pydantic model:
```python
class AttachFromDocumentPayload(BaseModel):
    document_id: uuid.UUID
    document_type: str = Field("other", pattern="^(resume|cover_letter|portfolio|certificate|other)$")

@router.post("/{app_id}/attachments/from-document", response_model=AttachmentOut)
def attach_from_document(
    app_id: uuid.UUID,
    payload: AttachFromDocumentPayload,
    ...
):
```

## SCAN 4: File Upload Validation Gaps

### Finding 4.1 — Document upload: no file size limit at router level (HIGH)

**File:** `backend/app/api/routers/documents/upload.py`

**Problem:** The upload endpoint reads the entire file into memory (`content = await file.read()` at ~L174) with **no size check before reading**. The size check only exists deep in the service layer (`crud_upload.py` L95: 50MB limit) **after** the content is already fully read into memory.

An attacker can upload a 500MB file. The server will:
1. Read the entire 500MB into RAM at L174
2. Perform MIME validation on 500MB in memory
3. Only later, the service layer would reject it

This is a memory exhaustion DoS vector.

**Severity:** HIGH  
**Fix:** Add an early size check before `file.read()`:
```python
# Check Content-Length header first (unreliable but fast reject)
if file.size and file.size > 50 * 1024 * 1024:
    raise HTTPException(status_code=413, detail="File too large. Max 50MB.")

content = await file.read()
if len(content) > 50 * 1024 * 1024:
    raise HTTPException(status_code=413, detail="File too large. Max 50MB.")
```
Better yet, stream chunks with a running counter (like `attachment_store.py` does).

### Finding 4.2 — Avatar upload: Content-Type checked but not verified (HIGH)

**File:** `backend/app/api/routers/auth/avatar.py` L105-113

**Problem:** The avatar endpoint validates `file.content_type.startswith('image/')` (L105). However, **`content_type` is a client-supplied header** — an attacker can upload a `.exe` file with `Content-Type: image/png` and it will pass validation.

No MIME magic verification is performed (unlike the document upload which uses `python-magic`). No file extension check. No actual image header validation.

The file isn't stored to disk in this endpoint (only the URL reference is saved), which limits the immediate impact. But if avatar serving is later implemented, malicious files would be served.

**Severity:** HIGH  
**Fix:** Add real content validation:
```python
# Verify actual content type
try:
    import magic
    detected = magic.from_buffer(data, mime=True)
    if not detected.startswith('image/'):
        raise HTTPException(400, "File content is not an image")
except ImportError:
    pass

# Also allowlist extensions
allowed_ext = {'.jpg', '.jpeg', '.png', '.gif', '.webp'}
ext = Path(file.filename or "").suffix.lower()
if ext not in allowed_ext:
    raise HTTPException(400, f"Image type {ext} not allowed")
```

### Finding 4.3 — Application attachment upload: no file type or size validation (HIGH)

**File:** `backend/app/api/routers/applications/attachments.py` L152-257

**Problem:** The `upload_attachment` endpoint:
1. **No file extension check** — any file type is accepted
2. **No content-type validation** — the file's MIME type is never checked
3. **No file size check** in the router — size is only checked in the underlying `attachment_store.py` (which streams in chunks)
4. **No filename sanitization** — only checks `file.filename` is truthy
5. `document_type` is a raw `str = Form("other")` with no validation against allowed values

Compare this to the document upload endpoint which has extension whitelists, MIME verification, and content validation. This endpoint has **none of those protections**.

**Severity:** HIGH  
**Fix:** Add the same validation as the document upload:
```python
ALLOWED_EXTENSIONS = {".pdf", ".docx", ".doc", ".txt", ".png", ".jpg", ".jpeg"}
ALLOWED_DOC_TYPES = {"resume", "cover_letter", "portfolio", "certificate", "other"}

if document_type not in ALLOWED_DOC_TYPES:
    raise HTTPException(400, f"Invalid document_type: {document_type}")

ext = Path(file.filename).suffix.lower()
if ext not in ALLOWED_EXTENSIONS:
    raise HTTPException(400, f"File type {ext} not allowed")
```

### Finding 4.5 — Document upload: MIME check silently skipped on error (MEDIUM)

**File:** `backend/app/api/routers/documents/upload.py` L215-225

**Problem:** If `python-magic` is installed but the MIME check throws an unexpected exception, the code catches it with a bare `except Exception` and **continues the upload** (`pass` on L225). This means a malformed file that crashes the MIME checker will be accepted.

```python
except Exception as mime_error:
    logger.error(...)
    pass  # <-- Upload proceeds despite MIME check failure
```

**Severity:** MEDIUM  
**Fix:** Reject the upload if MIME validation fails with an unexpected error:
```python
except ImportError:
    logger.warning("python-magic not available")
except Exception as mime_error:
    logger.error("MIME validation error", ...)
    raise HTTPException(400, "File content validation failed")
```

## SCAN 5: Open Redirects

### Finding 5.1 — OAuth error redirect uses `settings.FRONTEND_URL` (safe)

**File:** `backend/app/api/routers/auth/oauth.py` L106-107, L289

All redirect URLs are constructed using `settings.FRONTEND_URL` as the base, not from user input. The `error` query parameter is passed from Google's response but only as a query parameter value, not as a redirect destination.

```python
return RedirectResponse(f"{settings.FRONTEND_URL}/login?error={error or 'missing_code'}")
```

**Potential issue:** The `error` variable comes from Google's query parameter and is injected directly into the redirect URL without encoding. If `error` contains characters like `#` or `&`, it could manipulate the URL structure. However, since `FRONTEND_URL` is server-configured, this is low risk.

**Severity:** LOW (not a true open redirect)  
**Fix:** URL-encode the error parameter:
```python
from urllib.parse import quote
return RedirectResponse(f"{settings.FRONTEND_URL}/login?error={quote(error or 'missing_code')}")
```

### Finding 5.2 — No open redirects found

All other redirects in the codebase use hardcoded server-configured URLs (`settings.FRONTEND_URL`). No endpoint accepts a user-supplied URL and redirects to it.

**Status:** PASS
